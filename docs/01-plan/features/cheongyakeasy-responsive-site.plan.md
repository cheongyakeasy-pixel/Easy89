## 1. 개요

### 1.1 목적

청약이지는 사용자가 주택청약 공고, 접수 일정, 자격 확인 포인트, 준비 서류, 대출 관련 정보를 한 곳에서 쉽게 살펴볼 수 있도록 돕는 반응형 웹 사이트다. 첫 번째 버전은 로그인 없이 접근 가능한 정보형 서비스로 시작하며, 모바일과 데스크톱 모두에서 청약 정보를 빠르게 탐색하고 비교할 수 있게 만드는 것을 목표로 한다.

### 1.2 배경

주택청약 정보는 중요도가 높고 마감일이 명확하지만, 사용자가 실제로 확인해야 할 정보는 여러 공식 사이트와 공고문에 흩어져 있다. 사용자는 보통 다음 질문에 빠르게 답하고 싶어 한다.

- 내 지역에서 지금 신청 가능한 공고가 있는가?
- 접수 시작일과 마감일은 언제인가?
- 특별공급, 일반공급, 공공임대, 공공분양 중 어떤 유형인가?
- 내가 먼저 확인해야 할 자격 조건과 준비 서류는 무엇인가?
- 청약 전에 대출이나 자금 계획에서 주의할 점은 무엇인가?

청약이지는 이런 질문에 대한 첫 탐색 비용을 낮춘다. 단, 공식 신청이나 개인별 자격 판정까지 대신하지는 않고, 사용자가 공식 출처로 이동하기 전에 핵심 정보를 정리해 보는 보조 서비스로 설계한다.

## 2. 목표

### 2.1 주요 목표

- [ ] 사용자가 첫 화면에서 현재 접수 중이거나 접수 예정인 청약 공고를 확인할 수 있다.
- [ ] 지역, 공고 상태, 주택 유형, 공급 유형, 날짜 조건으로 공고를 필터링할 수 있다.
- [ ] 공고 상세 화면에서 일정, 자격 요약, 공급 정보, 준비 서류, 공식 출처 링크를 확인할 수 있다.
- [ ] 청약 접수 시작일, 마감일, 당첨 발표일, 계약 기간을 일정 화면에서 볼 수 있다.
- [ ] 청약 초보자를 위한 기본 개념, 자격 확인, 가점/순위, 서류 준비, 신청 절차 가이드를 제공한다.
- [ ] 주택청약과 함께 확인해야 할 대출/자금 준비 정보를 제공하되, 개인 금융 조언으로 보이지 않게 한다.
- [ ] Montage 디자인 시스템의 토큰과 컴포넌트 패턴을 적용해 안정적이고 현대적인 정보형 UI를 만든다.
- [ ] 이후 외부 API, 데이터 동기화, 즐겨찾기, 알림 기능으로 확장 가능한 데이터 구조를 준비한다.

### 2.2 비목표

- v1에서는 회원가입, 로그인, 개인별 맞춤 추천을 구현하지 않는다.
- v1에서는 청약 신청을 대행하지 않는다.
- v1에서는 사용자의 개인정보, 소득, 자산, 신용정보를 수집하지 않는다.
- v1에서는 법적 자격 판정이나 금융 상품 추천을 제공하지 않는다.
- v1에서는 청약홈, LH, SH, GH 등 공식 포털의 실시간 자동 수집 파이프라인을 필수로 만들지 않는다.
- v1에서는 SMS, 카카오톡, 이메일 등 외부 알림 발송을 구현하지 않는다.
- v1에서는 네이티브 모바일 앱을 만들지 않는다.

## 5. 성공 기준

### 5.1 Definition of Done

- [ ] 홈, 청약 공고 목록, 공고 상세, 청약 일정, 청약 가이드, 대출 정보 페이지가 구현되어 있다.
- [ ] React Router 기준으로 모든 주요 경로에 접근할 수 있다.
- [ ] seed data 기반 공고 목록과 상세 화면이 렌더링된다.
- [ ] URL query 기반 필터가 동작한다.
- [ ] 필터 결과 없음과 상세 공고 없음 상태가 구현되어 있다.
- [ ] 청약 일정 이벤트가 날짜순으로 표시된다.
- [ ] 공고 상세에 공식 출처 링크와 확인일이 표시된다.
- [ ] Montage-inspired 토큰 또는 WDS 토큰 adapter가 스타일에 반영되어 있다.
- [ ] 모바일 하단 내비게이션과 데스크톱 상단 내비게이션이 구현되어 있다.
- [ ] `npm run build`가 성공한다.
- [ ] 주요 viewport에서 브라우저 검증이 완료된다.

### 5.2 품질 기준

- [ ] 360px 모바일 화면에서 버튼, 카드, 필터 텍스트가 넘치지 않는다.
- [ ] 주요 버튼과 링크는 목적이 명확한 한국어 문구를 사용한다.
- [ ] 색상만으로 상태를 구분하지 않고 텍스트 배지를 함께 사용한다.
- [ ] 금액과 날짜는 tabular numeric 스타일을 적용한다.
- [ ] 대출 정보는 개인 금융 조언이 아닌 확인 가이드로 표현한다.
- [ ] 빈 상태 문구는 `데이터가 없습니다`처럼 차갑게 쓰지 않고, 사용자가 다음 행동을 이해하게 한다.
- [ ] 공식 출처 링크에는 `target="_blank"`와 `rel="noopener noreferrer"`를 적용한다.

## 8. 아키텍처 고려사항

### 8.1 프로젝트 레벨 선택

| 레벨 | 특징 | 선택 |
|------|------|:----:|
| Starter | 정적 사이트, 단순 구조, 빠른 학습/검증 | 부분 선택 |
| Dynamic | 기능별 모듈, 구조화된 데이터, 향후 API/BaaS 확장 | 선택 |
| Enterprise | 엄격한 계층 분리, MSA, 운영/보안 중심 | 미선택 |

현재 저장소는 빈 프로젝트에서 시작하므로 구현 난이도는 Starter에 가깝다. 다만 공고 데이터, 필터, 일정, 가이드, 대출 정보를 분리하고 향후 API 전환을 고려해야 하므로 문서와 구조는 Dynamic 기준으로 잡는다.

### 8.2 핵심 기술 결정

| 결정 항목 | 선택 | 이유 |
|-----------|------|------|
| 프레임워크 | Vite + React + TypeScript | 빠른 초기화, 정적 배포, 컴포넌트 기반 구현에 적합하다. |
| 라우팅 | React Router | 클라이언트 라우팅과 URL query 기반 필터를 단순하게 구현할 수 있다. |
| 데이터 | TypeScript seed data | v1 검증 속도를 높이고 타입 안정성을 확보한다. |
| 스타일링 | CSS variables + 일반 CSS | Montage 토큰을 명확히 반영하고 의존성을 줄인다. |
| 디자인 시스템 | Montage-inspired tokens and patterns | 공고 탐색, 필터, 카드, 내비게이션 같은 정보형 UI에 잘 맞는다. |
| 아이콘 | lucide-react 우선 | 설치와 사용이 쉽고 SVG currentColor 규칙을 유지하기 좋다. |
| 상태 관리 | URLSearchParams 중심 | 필터 공유, 뒤로가기, 새로고침 유지가 자연스럽다. |
| 테스트 | Vitest + 브라우저 검증 | 서비스 로직과 핵심 화면 흐름을 가볍게 검증한다. |

## 11. 디자인 방향

### 11.1 Montage 적용

청약이지의 시각 시스템은 Montage 적용 가이드를 우선한다.

- Primary: `#0066FF`
- Primary subtle: `#EAF2FE`
- Background: `#ffffff`
- Background alternative: `#F7F7F8`
- Text normal: `#171719`
- Line solid: `#E1E2E4`
- 기본 폰트: Pretendard
- breakpoint: `sm 768px`, `md 992px`, `lg 1200px`, `xl 1600px`

### 11.2 청약 정보 표현 원칙

- 공고와 일정은 정확한 날짜를 표시한다.
- 금액과 세대수는 임의 축약하지 않는다.
- 상태는 색상과 텍스트를 함께 표시한다.
- 주요 CTA는 `공식 공고 보기`, `공고 자세히 보기`, `필터 적용`처럼 행동이 분명해야 한다.
- 대출 관련 내용은 `확인`, `준비`, `주의` 중심으로 표현하고 개인별 추천처럼 보이지 않게 한다.
